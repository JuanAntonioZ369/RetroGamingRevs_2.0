COSAS QUE FALTAN PROBAR (en orden de probabilidad)
----------------------------------------------------

1. WINDOWS FIREWALL bloqueando RetroArch
   - Abrir "Seguridad de Windows" > Firewall > Permitir app
   - Agregar RetroArch-Win64\retroarch.exe como permitida (entrada Y salida)
   - O en PowerShell como admin:
     New-NetFirewallRule -DisplayName "RetroArch IN" -Direction Inbound -Program "C:\Users\jazc2\Desktop\NewGameRev\RetroArch-Win64\retroarch.exe" -Action Allow
     New-NetFirewallRule -DisplayName "RetroArch OUT" -Direction Outbound -Program "C:\Users\jazc2\Desktop\NewGameRev\RetroArch-Win64\retroarch.exe" -Action Allow

2. VER EL LOG EXACTO DE RETROARCH
   - Abrir RetroArch-Win64\retroarch.exe directamente (sin la app)
   - Settings > Logging > Log to File = ON, Log Level = Debug
   - Intentar hostear una sala online
   - Buscar el archivo retroarch_log.txt en RetroArch-Win64\
   - Copiar las lineas del error exacto

3. SINCRONIZAR retroarch.cfg con cfgIgual.txt
   - Siempre lanzar con run.bat, nunca con npm start directo
   - retroarch.cfg aun tiene valores viejos (check_frames=8, etc.)

4. COMPATIBILIDAD DEL SERVIDOR MITM
   - El binario /root/netplay-mitm-server/mitm puede ser una version vieja
   - Verificar version del RetroArch en uso vs version del mitm server
   - Probar bajando el codigo fuente y recompilando desde:
     https://github.com/libretro/netplay-mitm-server

5. PROBAR SIN ANUNCIO AL LOBBY
   - En retroarch.cfg cambiar temporalmente:
     netplay_public_announce = "false"
   - Si funciona sin anuncio → el problema es la comunicacion con lobby.libretro.com
   - Si sigue fallando → el problema es la conexion al MITM en si

6. PUERTO UDP — el MITM puede necesitar UDP ademas de TCP
   - El servidor solo escucha TCP (segun ss -tlnup)
   - Netplay de RetroArch usa UDP para el trafico de juego
   - Verificar si el binario mitm soporta UDP o solo TCP
